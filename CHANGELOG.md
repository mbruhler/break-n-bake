# Changelog

## 0.5.0 — 2026-04-17 — flattened slug layout + append-only validation layer

Two structural changes and two integration additions, all bundled because they share the same init/break flow.

### Changed (path flatten — BREAKING)
- **Runs live at `.bnb/<slug>/` instead of `.bnb/runs/<slug>/`.** The `runs/` parent is gone. Slugs are addressed directly under `.bnb/`.
- **Reserved-slug denylist** in `scripts/slugify.sh`: `config`, `current_run`, `readme`, `runs`, `validation`, `snapshots`, `active-agent`, `validation-error`. Any generated slug matching one of these gets `-run` suffixed to prevent collision with project-level files.
- **`forbidden_write_patterns`** updated from `.bnb/runs/*/…` globs to `.bnb/*/…` globs.
- **`.gitignore` lines** updated from `.bnb/runs/*/…` to `.bnb/*/…`.
- **All scripts, agents, commands, README, SKILL** updated to the new layout.

### Added (append-only validation layer)
- **`<slug>/validation/` directory** with three append-only subdirectories: `eslint/`, `tests/`, `prompts/`. Each contains numbered (`NNN-*`) files seeded by Breaker at break time.
- **Breaker seeds the layer** with `001-*` files per subdirectory, covering four bars: architecture boundaries, contract shapes, landmine defense, out-of-scope fences. Brownfield seeds derive from scout-report signals; greenfield from prompt intent.
- **`scripts/validation-lock.sh`** hashes every file under `validation/` into `.snapshots/validation.lock`. Run by Breaker after seed, by Baker after any addition.
- **`guard-fixer-paths.sh` extended**: any Write/Edit to a path recorded in `validation.lock` is rejected (Exit 2) for every agent. New numbered files not yet in the lock are allowed. Like database migrations — additive only.
- **`snapshot-verify.sh` extended** to also verify `validation.lock` alongside the existing contract lock.
- **Baker may add new numbered files** for new surface area introduced by a milestone; cannot edit existing files. Must re-run `validation-lock.sh` (and `regen-eslint-overlay.sh` if adding under `eslint/`).
- **`run-validation.sh` extended** with three new phases: per-run eslint overlays (`val-eslint-*.log` per overlay), per-run test files (`val-test-*.log` per test), and a prompt manifest consumed by Validator.
- **Validator spawns a read-only sub-agent per `validation/prompts/*.md`** (LLM-as-judge). Returns structured JSON verdict. Any `fail` is a blocker. `programmatic_checks` section added to Validator's JSON output.
- **`bnb-validator` frontmatter** gains `tools: Agent` to enable the prompt sub-agents. Still no Write/Edit — artifacts authored via Bash heredoc.

### Added (IDE + CLAUDE.md integration)
- **`scripts/inject-claude-md.sh`** writes (or refreshes) a `<!-- BEGIN break-n-bake -->…<!-- END -->` block at the top of the project's `CLAUDE.md`. Idempotent — only content between markers is managed; rest of file is preserved. Run on every `/init`.
- **`scripts/regen-eslint-overlay.sh`** composes the active run's `validation/eslint/*.json` into a single `eslint.config.bnb.mjs` at the project root. Run on every `/init` (stub) and `/break` (full composition). User adds one line to their real eslint config — `import bnb from './eslint.config.bnb.mjs';` — to surface the rules in their IDE.
- **`eslint.config.bnb.mjs`** added to `forbidden_write_patterns` so fixer cannot touch it; regenerated by script, never hand-edited.

### Migration
No automatic migration from 0.4.x. If you have a project initialized at 0.4.x with a populated `.bnb/runs/`:
1. Back it up.
2. Move each `.bnb/runs/<slug>/` up to `.bnb/<slug>/` (or re-init and re-run `/break`).
3. Update `.bnb/config.json` `version` field to `0.5.0` and the `forbidden_write_patterns` array to the new `.bnb/*/…` globs (or delete the file and re-init).
4. Runs from 0.4.x do not have a `validation/` directory. Validator in 0.5.0 treats a missing `validation/` as "no programmatic checks configured" — safe, but you lose the new contract layer until you re-break.

## 0.4.0 — 2026-04-17 — run scoping under `.bnb/runs/<slug>/`

Each `/break-n-bake:break` invocation now creates a fresh run directory, so running break-n-bake twice in the same project no longer stomps on prior artefacts.

### Added
- **`.bnb/runs/<slug>/` layout**: each run gets its own `_PROMPT.md`, `scout-report.json`, `spec/`, `milestones/`, `quality/`, `validation-results/`, `.snapshots/`, `.active-agent`. Slug is inferred by the orchestrator from the working prompt (e.g. `redesign-dashboard`, `create-habits-app`) and suffixed with `-2`, `-3`, … on collision.
- **`.bnb/CURRENT_RUN`**: plain-text pointer to the active run slug. Every command/script resolves the active run via `$BNB_RUN_DIR` → `$BNB_RUN` → `.bnb/CURRENT_RUN`.
- **`scripts/resolve-run.sh`**: central helper sourced by all run-scoped scripts.
- **`scripts/slugify.sh`**: deterministic slug transform (lowercase, `[a-z0-9-]`, 40-char cap).
- **`scripts/init-run.sh`**: creates a per-run scaffold with collision suffixing, writes `CURRENT_RUN`.

### Changed
- **`scripts/init-bnb.sh`**: no longer creates per-run dirs. Writes `config.json` (with `forbidden_write_patterns` now using `.bnb/runs/*/…` globs), `README.md`, and the `runs/` parent only. `.gitignore` lines updated to `.bnb/runs/*/validation-results/raw/`, `.bnb/runs/*/.snapshots/`, `.bnb/runs/*/.active-agent`.
- **`scripts/snapshot-lock.sh` / `snapshot-verify.sh` / `run-validation.sh` / `progress-check.sh`**: all resolve the active run via `resolve-run.sh` and write to `<run-dir>/.snapshots/` or `<run-dir>/validation-results/`.
- **`scripts/fixer-lock-on.sh` / `fixer-lock-off.sh`**: marker written both at `<run-dir>/.active-agent` and project-level `.bnb/.active-agent` (guard still reads the project-level marker).
- **`commands/break.md`**: infers a slug from the prompt, calls `init-run.sh`, writes `CURRENT_RUN`, passes run dir to Explorer and Breaker. Recap now includes the inferred slug.
- **`commands/bake.md` / `fix.md`**: every path references `$RUN_DIR = .bnb/runs/<slug>/` instead of `.bnb/…`. Preflight recap includes active-run line. Active run is frozen at command start; mid-flight `CURRENT_RUN` changes are ignored until next invocation.
- **`agents/bnb-{explorer,breaker,baker,validator,fixer}.md`**: each agent receives the run dir from orchestrator and writes only under that run. Forbidden-path list updated to `.bnb/runs/*/…`. Breaker's scaffold diagram updated.

### Migration
No automatic migration. If you have a project initialised at `0.3.x` with a populated `.bnb/`:
1. Back it up.
2. Move existing run-scoped dirs under `.bnb/runs/<slug>/`, or re-init the project and re-run `/break`.
3. Update `.bnb/config.json` version field (optional — not read).

## 0.3.0 — 2026-04-17 — prompt adherence layer

Applied four prompting techniques from the instruction-following literature across every prompt in the plugin (5 agents, 4 commands, 1 skill):

### Added
- **Rereading + recap gates** (Xu et al. 2023): every agent and command outputs a short numeric recap before any write/edit/agent-spawn. Forces read-in-full and confirms arguments parsed correctly instead of skimming.
- **`<output_format>` blocks** with exact templates: `bake-summary` (Baker), `bake-report` (/bake), `break-report` (/break), `fix-report` (/fix), `init-report` (/init), `activation-message` (skill). Evidence field is mandatory — deliverables without evidence are not done.
- **`<hard_rules>` XML blocks**: rule lists separated from prose prompts (Anthropic best-practice tag separation), making rules more recall-stable mid-conversation.
- **End-of-prompt `<reminder>` blocks** (Liu et al. 2023 "Lost in the Middle" mitigation): 3–5 point verification checklists repeated at end of each file, addressing critical rules that would otherwise only appear mid-prompt.
- **`CRITICAL` / `IMPORTANT` markers**: `CRITICAL` on rules whose violation corrupts the contract (snapshot-verify, `--no-auto-fix` gate, `_PROMPT.md` verbatim, one-milestone-per-invocation, no edits to spec/quality); `IMPORTANT` on rules whose violation is costly but recoverable (style, surgical diffs, strict lint).

### Scope
Files touched: `skills/break-n-bake/SKILL.md`, `agents/bnb-{explorer,breaker,baker,validator,fixer}.md`, `commands/{init,break,bake,fix}.md`. No runtime or script changes.

### Rationale
Previous versions had rules; they were being ignored mid-flow. The symptoms: Baker drifting into M+1 without OK, verdicts silently re-classified, `> Ask user if...` lines decided unilaterally, forbidden-path edits. Root cause was prompt shape, not missing rules. Four techniques, one layer, no prose bloat.

## 0.2.0 — 2026-04-17 — raise minimum model to Sonnet 4.6

### Changed
- **`bnb-explorer`**: `model: haiku` → `model: sonnet` with `effort: medium`. Scout's `break vs direct` recommendation is load-bearing — a Haiku miss here skips the whole structured workflow. Sonnet with medium thinking reads the prompt and repo well enough to make the judgment reliably.
- **`bnb-validator`**: `model: haiku` → `model: sonnet` with `effort: medium`. Severity classification (blocker vs deferrable) requires reading spec and import graph to judge whether a failing test breaks a contract or just an edge case. Haiku guessed; Sonnet actually reads.

### Rationale
Haiku was tempting for cost but both affected agents make judgment calls, not lookups. The cost of one falsely-`clean` milestone that propagates into M+2 is larger than the marginal spend of running Sonnet on reconnaissance and classification.

### Unchanged
- `bnb-breaker` stays Opus (partitioning and cross-refs are the heaviest cognitive step).
- `bnb-baker` stays Sonnet default, Opus for `risk: high` milestones.
- `bnb-fixer` stays Opus (root-cause work on blockers).

## 0.1.0 — 2026-04-17 — initial release

First public version. MVP scope.

### Included
- Manifest with `userConfig` for `max_fix_iterations` and `break_threshold_files`.
- Five sub-agents with explicit model assignments:
  - `bnb-explorer` (haiku) — prompt + repo reconnaissance
  - `bnb-breaker` (opus) — produces `.bnb/spec/`, `milestones/`, `quality/`
  - `bnb-baker` (sonnet) — implements one milestone per invocation
  - `bnb-validator` (haiku, `disallowedTools: Write, Edit`) — validation + severity classification
  - `bnb-fixer` (opus) — repairs blockers; blocked from test/config paths
- Four commands: `/init`, `/break`, `/bake`, `/fix`
- Orchestrator skill auto-suggesting break on long prompts, refactor keywords, or large blast radius
- Hook-based path guard for fixer: `SubagentStart/Stop` markers + `PreToolUse` Write/Edit filter
- SHA256 snapshot lock + verify as belt-and-suspenders over the hook
- Stack autodiscovery: node, python, rust, go, ruby, php, jvm
- Per-milestone validation runs with structured JSON + human summary
- Bounded fix cycle: configurable max iterations, hard stop at 3 consecutive no-progress iterations

### Known limitations
- `SubagentStart/Stop` matcher behavior is documented but runtime-specific; snapshot-verify runs as a second line of defense.
- Validation command inference for non-JS/TS stacks is best-effort; users may need to tweak `.bnb/config.json` after init.
- End-to-end workflow has not yet been run against a real large-refactor scenario.
